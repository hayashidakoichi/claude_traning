# Claude Code 実践トレーニング（２日目） — Skills・Hooks・サブエージェント

## はじめに

前回のトレーニングで Claude Code のインストールと基本操作を習得した。２日目は Claude Code をさらにパワーアップさせる **Skills（スキル）**・**Hooks（フック）**・**SubAgents（サブエージェント）** を学ぶ。

まず座学で「それぞれ何なのか」「どう使い分けるのか」を理解し、後半のハンズオンで実際に体験する。

### 先に結論：どの機能を使えばいいの？

Claude Code にはいくつかの拡張機能がある。迷ったら以下の判断フローで選ぶ。

```
特定イベント時に確実実行したい？     → Hooks
常に必要な知識？                     → CLAUDE.md
独立タスクで、結果のみ必要？         → サブエージェント
繰り返し手動で呼び出す？             → スラッシュコマンド
その他の専門知識？                   → スキル
```

> 参考: [Claude Codeの機能使い分けガイド](https://zenn.dev/notahotel/articles/a175aa95629d0b)

## セクション 1：Skills（スキル）とは何か？（座学 15 分）

![Claude Skill](https://res.cloudinary.com/zenn/image/fetch/s--gTZdvrku--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://storage.googleapis.com/zenn-user-upload/deployed-images/8d97d8f90eb5cf1a4d870e4b.png%3Fsha%3D69c734afb9784db266de2d98368c57aa56fa61e6)
*出典: [Claude CodeにおけるClaude SkillsとSubAgentsの使い分けと併用を理解する](https://zenn.dev/nogu66/articles/claude-code-think-abount-skills-and-subagent)*

### 1-1. スキルの定義

**Skills は、Claude に特定分野の専門知識を提供する「再利用可能な学習教材」。**

公式の定義では「Claude がタスクに関連したタイミングで動的に検出・読み込みする指示、スクリプト、リソースを含むフォルダ」とされている。Excel スプレッドシートの操作から組織のブランドガイドラインの遵守まで、特定の分野における専門知識を Claude に提供する、**専門的なトレーニングマニュアル**のようなもの。

> 参考: [Skills, Explained — Claude Blog](https://www.claude.com/blog/skills-explained)

### 1-2. たとえ話：スキルは「必要な時だけ開くレシピ本」

普段の料理を想像してみてほしい。

あなたは普段からよく料理をしている。当然、レシピ本は棚にしまってある（メモリ消費ゼロ）。常にレシピ本を開いているわけではない。

そして「カルボナーラを作りたい」と思った時だけ、イタリアンのレシピ本を取り出して、カルボナーラのページだけを開く。

**これが Skills。**

仮に CLAUDE.md（後述）にすべての情報を書き込んでしまうと、レシピ本を常に全ページ開いた状態で料理するようなもの。テーブルの上が散らかって、肝心の情報が見つけにくくなる。

Skills を使うと、**必要な時だけ必要なページを開く**ので、効率的に作業できる。

> 参考: [【Claude Code】SkillsとSubAgents、どっちを使えばいいの？](https://zenn.dev/tmasuyama1114/articles/claude_code_skills_vs_subagents) — とまだ@AI 駆動開発教育者

### 1-3. スキルの主な特徴

| 特徴 | 説明 |
| --- | --- |
| **再利用可能** | 複数の会話やプロジェクト全体で共有できる |
| **段階的読み込み** | 必要な時だけ詳細を読み込む（Progressive Disclosure） |
| **専門知識の保持** | 組織のガイドライン、ドメイン専門知識などを保存 |

### 1-4. スキルのフォルダ構成

```
my-skill/
├── SKILL.md          # 必須: 指示 + メタデータ
├── scripts/          # 任意: スクリプト
├── references/       # 任意: リファレンス
└── assets/           # 任意: テンプレートやリソース
```

**SKILL.md のみが必須**（SKILL.md だけでもスキルとして機能する）。

### 1-5. SKILL.md の中身

```yaml
---
name: my-skill
description: このスキルの説明。Claudeがいつ使うべきかも書く。
---
# ここに自然言語で指示を書く
```

先頭のフロントマターに `name`（スキル名）と `description`（使用時機）を記載。本文は Markdown で自由に指示を記述する。

#### 具体例：コードレビュースキル

```yaml
---
name: code-review
description: diffやコミット、PRをレビューする際に使用してください。
---

- 日本語で応答
- 重大な問題にフォーカス
- suggestionブロックを使用
```

> 参考: [Claude Codeの機能使い分けガイド](https://zenn.dev/notahotel/articles/a175aa95629d0b)

### 1-6. 段階的情報開示（Progressive Disclosure）

スキルの読み込みは 3 段階で行われる。これにより、コンテキストウィンドウを圧迫しない設計になっている。

![スキルの読み込み](https://res.cloudinary.com/zenn/image/fetch/s--hUIbLTvH--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://storage.googleapis.com/zenn-user-upload/deployed-images/54c031387b1851c0a7f1dc60.png%3Fsha%3D27cd4d6e6b1fad3fdb9ae8bdb3e22bda46852220)
*出典: [Claude CodeにおけるClaude SkillsとSubAgentsの使い分けと併用を理解する](https://zenn.dev/nogu66/articles/claude-code-think-abount-skills-and-subagent)*

1. **全スキルの name / description を確認**（軽量 — 名前だけチラ見する）
2. **関連ありと判断したスキルの SKILL.md 本文をロード**（レシピ本を棚から取り出す）
3. **必要に応じて references/ 等の追加ファイルを読み込み**（該当ページを開く）

> **ポイント**: エージェント側には「スキルの名前」しか置かず、必要な時だけ中身を読み込む。これにより無駄なコンテキスト消費を防ぐ。

### 1-7. コンテキスト効率の具体的な数値

Progressive Disclosure がどれだけ効率的か、数値で見てみる。

```
【従来の方法】全てをメインプロンプトに含める
- 専門知識A: 5,000 トークン
- 専門知識B: 5,000 トークン
- 専門知識C: 5,000 トークン
合計: 15,000 トークン（常に消費）

【Skills を使う方法】必要な時だけ読み込む
- Skill メタデータ × 3: 300 トークン（常に消費）
- 必要な Skill の詳細: 5,000 トークン（使う時だけ）
平均消費: 5,300 トークン（約 65% 削減）
```

> 参考: [Claude CodeにおけるClaude SkillsとSubAgentsの使い分けと併用を理解する](https://zenn.dev/nogu66/articles/claude-code-think-abount-skills-and-subagent)

### 1-8. スキルの置き場所

| スコープ | パス |
| --- | --- |
| ユーザー（個人） | `~/.claude/skills/<skill-name>/SKILL.md` |
| プロジェクト（チーム共有） | `.claude/skills/<skill-name>/SKILL.md` |

### 1-9. スキルの動作フロー

ドキュメントテンプレート集スキルで「手順書を書いて」と依頼した場合：

1. SKILL.md の description 基準でスキル発火
2. SKILL.md 本文を読み込み
3. リファレンスにあるテンプレートファイルを読み込み
4. テンプレートに沿った手順書を生成

### 1-10. スキルの具体的な活用シーン

| シーン | 説明 |
| --- | --- |
| **デザインシステム** | 普段のロジック実装時には不要だが、UI を作る時だけ「当社のデザインガイドライン」を読み込ませる |
| **社内テンプレート** | 週次レポートや議事録のフォーマットを統一する |
| **ブランドガイドライン** | プレゼン資料作成時に自社カラーやフォントを自動適用する |

> 参考: [【Claude Code】SkillsとSubAgents、どっちを使えばいいの？](https://zenn.dev/tmasuyama1114/articles/claude_code_skills_vs_subagents)

## セクション 2：Hooks（フック）とは何か？（座学 10 分）

### 2-1. フックの定義

**Hooks は「A が起こったら B する」という自動処理を設定する仕組み。**

`.claude/settings.json` に JSON で設定し、特定のイベント（ファイル保存、セッション終了など）が発生した時に、決められたコマンドを自動実行する。

> 参考: [Claude Code Hooks Guide](https://code.claude.com/docs/en/hooks-guide)

### 2-2. たとえ話：フックは「家のセンサーライト」

玄関にセンサーライトを設置している家を想像してほしい。

- **人が通ったら**（イベント）→ **ライトが点灯する**（自動処理）

毎回スイッチを押す必要はない。センサーが検知して、自動で反応する。

**これが Hooks。**

Claude Code の世界では：
- **コードを変更したら** → **自動でフォーマッタを実行する**
- **セッションが終了したら** → **通知音を鳴らす**
- **ファイルを保存したら** → **自動でテストを走らせる**

といった「イベント駆動の自動処理」を設定できる。

### 2-3. フックの設定例

#### 例 1：通知時に音を鳴らす

```json
{
  "hooks": {
    "Notification": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "afplay /path/to/Ping.aiff &"
          }
        ]
      }
    ]
  }
}
```

#### 例 2：セッション終了時に CLAUDE.md の更新を提案する

前回のトレーニングで紹介した CLAUDE.md 自動更新の仕組みも、Hooks で実現している。

```
セッション終了（イベント）
  ↓
Hook が自動でスクリプトを実行（自動処理）
  ↓
会話履歴を分析し、CLAUDE.md への追記提案を生成
```

> 参考: [Claude Codeの機能使い分けガイド](https://zenn.dev/notahotel/articles/a175aa95629d0b)

### 2-4. フックを使うべき場面

| 場面 | 例 |
| --- | --- |
| **特定のタイミングで確実に処理を実行したい** | コード変更時にフォーマッタ実行 |
| **人間が忘れがちな処理を自動化したい** | コミット前にテスト実行 |
| **イベントに応じた通知が欲しい** | 処理完了時に音やメッセージで通知 |

### 2-5. Skills との違い

| 項目 | Skills | Hooks |
| --- | --- | --- |
| たとえ | レシピ本 | センサーライト |
| 発火条件 | Claude が「関連あり」と判断した時 | 特定イベント発生時（確実に実行） |
| 用途 | 専門知識の提供 | 自動処理の実行 |
| 設定場所 | `.claude/skills/` | `.claude/settings.json` |

> **ポイント**: Skills は「Claude が判断して使う」のに対し、Hooks は「イベントが起きたら確実に実行される」。確実性が求められる処理には Hooks を使う。

## セクション 3：SubAgents（サブエージェント）とは何か？（座学 15 分）

![SubAgent](https://res.cloudinary.com/zenn/image/fetch/s--ZVOGIvVd--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://storage.googleapis.com/zenn-user-upload/deployed-images/1e5be3a4e1128e03e997fc01.png%3Fsha%3Dda7809c13c05048004dbc139c116b764ed4cd648)
*出典: [Claude CodeにおけるClaude SkillsとSubAgentsの使い分けと併用を理解する](https://zenn.dev/nogu66/articles/claude-code-think-abount-skills-and-subagent)*

### 3-1. サブエージェントの定義

**SubAgents は、独立したコンテキストで特定タスクを処理する「専門家ユニット」。**

独自のコンテキストウィンドウ、カスタムシステムプロンプト、特定のツール権限を持つ専門的な AI アシスタント。Claude Code や Claude エージェント SDK で利用でき、個別のタスクを独立して処理する。

### 3-2. たとえ話：サブエージェントは「優秀なアシスタント」

あなたはプロジェクトマネージャーとして、全体の設計を考えている（メインの Claude）。

そこで、少し気になる問題が見つかった。ただし、自分で大量の資料を読み漁ると、今の設計作業の集中力が切れてしまう。

そこで、アシスタント（SubAgent）にこう頼む。

**「この問題の原因を突き止めておいて。わかったら結果だけ教えて」**

するとアシスタントは大量の資料を読み漁り、原因を特定する。そして、メインの Claude に **「原因はこれでした」と報告だけを持って帰ってくる**。

それを受けて、メインの Claude はまた本筋の作業に戻ることができる。

**これが SubAgents。**

> 参考: [【Claude Code】SkillsとSubAgents、どっちを使えばいいの？](https://zenn.dev/tmasuyama1114/articles/claude_code_skills_vs_subagents) — とまだ@AI 駆動開発教育者

### 3-3. サブエージェントの主な特徴

| 特徴 | 説明 |
| --- | --- |
| **独自の権限** | メインエージェントと異なるツール権限を設定可能 |
| **独立実行** | 個別のコンテキストで独立して動作 |
| **並列処理** | 複数のサブエージェントが同時にタスクを実行可能 |

### 3-4. サブエージェントの実行環境

![独立したタスク実行環境](https://res.cloudinary.com/zenn/image/fetch/s--nq8QkCJ9--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://storage.googleapis.com/zenn-user-upload/deployed-images/352f37227a6acd9d2bfc726b.png%3Fsha%3D5ea3a6278bd87aac0e43c384085bb97b64a23c14)
*出典: [Claude CodeにおけるClaude SkillsとSubAgentsの使い分けと併用を理解する](https://zenn.dev/nogu66/articles/claude-code-think-abount-skills-and-subagent)*

サブエージェントの「独立した環境」が強み。メインの会話履歴にはアクセスせず、結果のみがメイン会話に返される。そこで Skills を呼び出すことで効果が最大化される。

### 3-5. サブエージェントの定義ファイル

`.claude/agents/{agent-name}.md` で定義する。

#### 例：翻訳エージェント

```yaml
---
name: translator
description: 既存翻訳との整合性を保ちながら日本語を英語に翻訳します。
---

あなたは翻訳の専門家です。

1. 翻訳対象を解析する
   - 同じ訳や似た文章の既存調査
   - 文脈を調査
2. 翻訳する
3. 翻訳結果の妥当性を検証する
```

> 翻訳過程の膨大なコンテキスト消費を子エージェントに隔離でき、親は結果だけを記憶する。

> 参考: [Claude Codeの機能使い分けガイド](https://zenn.dev/notahotel/articles/a175aa95629d0b)

### 3-6. `context: fork` でサブエージェント実行

スキルの frontmatter に `context: fork` を付けると、スキルがサブエージェントとして隔離実行される。メインの会話コンテキストを汚さない。

```yaml
---
name: deep-research
description: Research a topic thoroughly
context: fork
agent: Explore
---
Research $ARGUMENTS thoroughly:
1. Find relevant files using Glob and Grep
2. Read and analyze the code
3. Summarize findings with specific file references
```

### 3-7. サブエージェントの具体的な活用シーン

| シーン | 説明 |
| --- | --- |
| **大規模な調査** | プロジェクトの全ファイルから特定のキーワードを洗い出す |
| **ドキュメントの読み込み** | 新しい資料を丸ごと読ませて「要点をまとめて」と依頼する |
| **実装の検証** | 「今の内容が正しいか、客観的な視点でチェックして」と依頼する |
| **並列処理** | 市場調査・競合分析・トレンド分析を 3 つ同時に実行する |

> 参考: [【Claude Code】SkillsとSubAgents、どっちを使えばいいの？](https://zenn.dev/tmasuyama1114/articles/claude_code_skills_vs_subagents)

## セクション 4：Skills・Hooks・SubAgents の使い分け（座学 10 分）

### 4-1. 一言で言うと

| 機能 | たとえ | 役割 |
| --- | --- | --- |
| **Skills** | 必要な時だけ開くレシピ本 | 「やり方」を教える |
| **Hooks** | 家のセンサーライト | 「自動で実行」する |
| **SubAgents** | 優秀なアシスタント | 「仕事」を任せる |

### 4-2. 判断フローチャート

迷った時はこのフローで判断する。

![スラッシュコマンド](https://storage.googleapis.com/zenn-user-upload/fe40e8f87e33-20260116.png)
*出典: [Claude Codeの機能使い分けガイド](https://zenn.dev/notahotel/articles/a175aa95629d0b)*

**第 1 の質問**: 「特定のタイミングで確実に実行したい処理ですか？」

```
YES → Hooks を使う
（例：コード変更時にフォーマッタ実行、セッション終了時に通知）
```

**第 2 の質問**: 「AI に守らせたいルールや知識ですか？」

```
YES → Skills を使う
（例：デザインルール、テンプレート、ブランドガイドライン）
```

**第 3 の質問**: 「大量の情報を処理したり、独立した作業が必要ですか？」

```
YES → SubAgents を使う
（例：大量資料の調査、バグ原因の特定、ドキュメントの要約）
```

> 参考: [【Claude Code】SkillsとSubAgents、どっちを使えばいいの？](https://zenn.dev/tmasuyama1114/articles/claude_code_skills_vs_subagents)

### 4-3. 比較まとめ

| 特徴 | Skills（スキル） | Hooks（フック） | SubAgents（サブエージェント） |
| --- | --- | --- | --- |
| イメージ | 参考書、マニュアル | センサーライト | アシスタント、別働隊 |
| コンテキスト | 使う時だけ消費 | 消費しない | 独立（メインを汚さない） |
| 発火条件 | Claude の判断 | イベント発生時 | 明示的に指示 |
| 主な用途 | 知識の提供 | 自動処理 | 調査・探索・検証 |
| 設定場所 | `.claude/skills/` | `.claude/settings.json` | `.claude/agents/` |

### 4-4. Skills と SubAgents の併用パターン（推奨）

公式が推奨する最も効果的な使い方は **SubAgent から Skills を呼び出す** パターン。

![スキルとサブエージェントの併用](https://res.cloudinary.com/zenn/image/fetch/s--z-7t4_zj--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_1200/https://storage.googleapis.com/zenn-user-upload/deployed-images/8b41bde00707dde46181a6a3.png%3Fsha%3Da6c418f8c176ab39d90b13d7c90e489f11b4762a)
*出典: [Claude CodeにおけるClaude SkillsとSubAgentsの使い分けと併用を理解する](https://zenn.dev/nogu66/articles/claude-code-think-abount-skills-and-subagent)*

- SubAgent A（市場調査）→ Skill（分析フレームワーク）を呼び出し
- SubAgent B（競合分析）→ 同じ Skill（分析フレームワーク）を呼び出し
- SubAgent C（トレンド分析）→ 同じ Skill（分析フレームワーク）を呼び出し

> 同じ「分析フレームワーク」というスキルを 3 つのサブエージェントが共有して使っている。知識は Skills に集約し、実行は SubAgents に任せることで、再利用性と効率性を両立できる。

### 4-5. CLAUDE.md・MCP との違い

| 項目 | Skills | CLAUDE.md |
| --- | --- | --- |
| 役割 | 専門的タスクの「実行方法」を教える | プロジェクト固有の情報を Claude に伝える |
| スコープ | ポータブルな専門知識 | 特定リポジトリに紐づく |
| 読み込み | 必要な時だけ | 常に読み込まれる |

| 項目 | Skills | MCP Servers |
| --- | --- | --- |
| 役割 | データを「どう扱うべきか」を教える | 外部データソースへの「接続」を提供 |
| 例 | テンプレートに沿った文書作成 | GitHub や DB へのアクセスを可能にする |

> 参考: [Claude Code Skills (Agent Skills) 入門【社内勉強会スライド】](https://dev.classmethod.jp/articles/claude-code-skills-for-cloud-bu-consulting-members/)

## セクション 5：Skills 活用の Tips（座学 5 分）

### 5-1. SKILL.md は 500 行以下に保つ

SKILL.md はエントリーポイント。概要やナビゲーションとして機能させ、詳細は `references/` 等に分割する。

```
my-skill/
├── SKILL.md          # 概要 + ナビゲーション
├── references/       # 詳細リファレンス
├── examples.md       # 使用例
└── scripts/
    └── helper.sh
```

### 5-2. スキルが発火しないとき

`description` を見直す。Claude が「いつ使うべきか」を判断できる記述になっているか確認する。

最終手段は手動実行: `/skill-name` でスラッシュコマンドとして直接実行（例: `/fix-issue 123`）。

### 5-3. セキュリティに注意

スキルの入手元は信頼できるソースに限定する:

- 自作スキル
- 社内で管理・レビューされたスキル
- 信頼できる公開スキル（例: anthropics/skills）

第三者スキルを使う場合はスキルの中身をすべて確認する。判断できない場合は使わない。

### 5-4. スキルを作るスキル

公式の `skill-creator` を導入すると、スキルの作成・改善を対話的に進められる。

```
> /skill-creator
⏺ スキルクリエイターへようこそ！
  1. 新しいスキルを作成したい
  2. 既存のスキルを改善したい
  3. スキルのテスト・評価を実行したい
  4. スキルの説明文（トリガー）を最適化したい
```

> 参考: [Claude Code Skills (Agent Skills) 入門【社内勉強会スライド】](https://dev.classmethod.jp/articles/claude-code-skills-for-cloud-bu-consulting-members/)

## セクション 6：Skills 実践デモ（ハンズオン 40 分）

ここからは実際に Claude Code でスキルを体験する。各デモは講師がスクリーンに映しながら進行し、参加者も手元で同じ操作を行う。

### デモ ①「プレゼン作って」→ .pptx が生成される（10 分）

**使用スキル**: ビルトイン `pptx`
**狙い**: 「えっ、ファイルまで作れるの！？」

Claude Code にはプレゼンテーションファイル（.pptx）を生成するビルトインスキルが搭載されている。自然言語で指示するだけで、PowerPoint ファイルが完成する。

#### 手順

1. Claude Code を起動する

2. 以下のプロンプトを入力する:

```
「Claude Code 導入のメリット」というテーマで、5枚スライドのプレゼン資料を作ってください。
対象は非エンジニアの管理職です。
```

3. `.pptx` ファイルが生成されることを確認する

4. 生成されたファイルを PowerPoint やプレビューで開いて内容を確認する

> **ポイント**: テキストだけでなくファイルそのものを生成できる。これがスキルの力。

---

### デモ ②  ブランドガイドラインを適用する（10 分）

**使用スキル**: `brand-guidelines`
**狙い**: 「自社デザインが自動適用される！」

同じプレゼン生成でも、ブランドガイドラインスキルを使うと自社のカラーやフォント、ロゴの使用ルールが自動で適用される。

#### 事前準備：ブランドガイドラインスキルの作成

まず、自社のブランドガイドラインをスキルとして登録する。

```
以下の内容でブランドガイドラインのスキルを作成してください。

会社名: カラフルボックス
コーポレートカラー: #2E86AB（メインブルー）、#F5F5F5（背景グレー）
フォント: メイリオ
ロゴ使用ルール: スライドの右上に必ず配置、最小サイズ 40px
スライドのフッター: 「株式会社カラフルボックス Confidential」
```

> Claude Code が `.claude/skills/brand-guidelines/SKILL.md` を自動生成してくれる。

#### 手順

1. ブランドガイドラインスキルが作成されたことを確認する

2. デモ ① と同じプロンプトを再度入力する:

```
「Claude Code 導入のメリット」というテーマで、5枚スライドのプレゼン資料を作ってください。
対象は非エンジニアの管理職です。
```

3. 生成された `.pptx` を開き、ブランドガイドラインが反映されていることを確認する:
   - コーポレートカラーが使われているか
   - フッターが入っているか

> **ポイント**: スキルを登録しておけば、毎回指示しなくても自社デザインが自動適用される。「description」に「プレゼン資料やドキュメントを作成する際に使用」と書いておくことで自動発火する。

### デモ ③「週次レポート書いて」（10 分）

**使用スキル**: `internal-comms`（社内コミュニケーション用スキル）
**狙い**: 「テンプレ通りに出てくる！」

社内の週次レポートなど、決まったフォーマットがある文書もスキルで自動化できる。

#### 事前準備：社内コミュニケーションスキルの作成

```
以下の内容で社内コミュニケーション用のスキルを作成してください。

スキル名: internal-comms
用途: 週次レポートや社内報告書を、社内テンプレートに沿って作成する

週次レポートのテンプレート:
# 週次レポート（YYYY/MM/DD）
## 今週の成果
- （箇条書きで 3〜5 項目）
## 来週の予定
- （箇条書きで 3〜5 項目）
## 課題・相談事項
- （あれば記載）
## 数値報告
| 指標 | 今週 | 先週 | 増減 |
```

#### 手順

1. スキルが作成されたことを確認する

2. 以下のプロンプトを入力する:

```
今週の週次レポートを書いてください。
成果: Claude Code のトレーニング資料作成、社内勉強会の実施
来週: トレーニング第2回の実施、活用事例の収集
課題: 参加者のスキルレベルにばらつきがある
```

3. テンプレートに沿ったレポートが出力されることを確認する

> **ポイント**: スキルにテンプレートを登録しておけば、毎回フォーマットを指定する必要がない。新しいメンバーでも統一されたフォーマットで報告書を作成できる。

### デモ ④「自分の業務用スキルを作ろう」（10 分）

**使用スキル**: `skill-creator`
**狙い**: 「自分でも作れるんだ」

公式の `skill-creator` スキルを使って、参加者それぞれが自分の業務に合ったスキルを作成する。

#### 手順

1. Claude Code で以下を入力する:

```
/skill-creator
```

2. スキルクリエイターが起動したら、対話形式で進める:

```
⏺ スキルクリエイターへようこそ！
  1. 新しいスキルを作成したい       ← これを選ぶ
  2. 既存のスキルを改善したい
  3. スキルのテスト・評価を実行したい
  4. スキルの説明文（トリガー）を最適化したい
```

3. 自分の業務に合ったスキルを考えて作成する

#### スキルのアイデア例

| 業務 | スキル案 |
| --- | --- |
| 採用業務 | 求人票テンプレート自動生成 |
| 給与計算 | 給与明細の確認チェックリスト生成 |
| 社内研修 | 研修アンケートの集計レポート作成 |
| 顧客対応 | FAQ 回答テンプレート生成 |
| 営業支援 | 提案書の構成案自動生成 |

4. 作成されたスキルの SKILL.md を確認する:

```
作成されたスキルの中身を見せてください
```

5. 実際にスキルを使ってみる:

```
（作成したスキルのトリガーとなる依頼を入力する）
```

> **ポイント**: スキルは「自分の仕事のやり方を Claude に教える」こと。プログラミングの知識は一切不要。日本語で書くだけ。

## セクション 7：まとめと振り返り（10 分）

### 本日学んだこと

| テーマ | 内容 |
| --- | --- |
| Skills の基本 | レシピ本のたとえ・段階的情報開示・65% のコンテキスト削減 |
| Hooks の基本 | センサーライトのたとえ・イベント駆動の自動処理 |
| SubAgents の基本 | 優秀なアシスタントのたとえ・独立実行環境・並列処理 |
| 使い分け | Skills = 知識、Hooks = 自動処理、SubAgents = 仕事の委任 |
| 実践 | pptx 生成 → ブランド適用 → テンプレ活用 → 自作スキル |

### スキル活用のステップ

```
Step 1: まずビルトインスキルを使ってみる（pptx, xlsx など）
    ↓
Step 2: 自社ルールをスキルとして登録する（ブランド、テンプレート）
    ↓
Step 3: 業務ごとの専門スキルを作成する（skill-creator）
    ↓
Step 4: チームでスキルを共有し、ナレッジを蓄積する
```

### 宿題

次回までに、自分の業務で「毎回同じ指示を出している作業」を 1 つ見つけて、スキル化を試みてください。
