# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## skills

- ニュースレターを作る時には skillsの"internal-comms"を使って下さい。

## リポジトリの性格

このリポジトリはソフトウェアプロジェクトではなく、**ビジネスユーザー向け Claude Code ハンズオン勉強会の教材・提案書一式**を管理する Markdown 中心のドキュメントリポジトリです。ビルド・テスト・リントのコマンドはありません。`node_modules/` は存在しますが過去の遺物で、package.json は無く開発ワークフローには使われていません。

編集対象は原則として Markdown であり、受講者（非エンジニアの中小企業ユーザー）が読みやすい日本語・用語・粒度で書くことが最優先です。

## ディレクトリ構成の意味

ファイル名と配置には役割分担があり、これを踏まえないと「どこを直せばいいか」が分からなくなります。

- **ルート直下の `*.md`** — 受講者に配布する正式な教材。README.md の表がマスター。新しい教材ファイルを追加・リネームしたら README.md の表を必ず更新する（過去に齟齬が起きたため）。
  - `ClaudeCode勉強会.md` — 座学資料
  - `実践トレーニング.md` — 1日目ハンズオン
  - `実践トレーニング（２日目）_参考資料.md` — 2日目参考資料
  - `Claude_Code_Skills_実践トレーニング.md` — Skills ハンズオン本体（現在の主な編集対象）
  - `Claude_code_Skills_実践トレーニング_ハンズオン2_サンプル議事メモ.md` — ハンズオン用サンプル入力データ
- **`proposal/`** — クライアント（カラフルボックス社など）向けの提案書。`.gitignore` に載っているためコミット対象外。
- **`refs/`** — 教材内で参照する画像・PDF 等の素材置き場（スクリーンショット、サンプル PDF 等）。
- **`workspace/`** — **ハンズオン実行時の出力先サンドボックス**。`.gitignore` 済み。教材プロンプトに従って生成したニュースレター・議事録・pptx・コピー等はここに置く。ルートや教材ファイル自体に混ぜない。
- **`old/`, `旧トレーニングメニュー/`** — 旧版。`.gitignore` 済み。参照はしても編集・削除は依頼がない限りしない。
- **`.claude/skills/`** — プロジェクトローカルの Skill。公式 `example-skills` プラグインキャッシュからコピーしてきた `internal-comms` が入っており、教材のハンズオンで「プロジェクト配下に置いた Skill が発火する」ことを示すために使われている。Skill を追加する場合は同じ `SKILL.md` + `examples/` 構造で置く（教材本文でこの構造自体を解説しているため、崩すと資料と不整合になる）。
- **`.claude/settings.local.json`** — 個人のローカル設定。編集しない。

## 教材を編集するときの注意

- **`.gitignore` されたディレクトリ（`proposal/`, `workspace/`, `old/`, `旧トレーニングメニュー/`, `.claude/`）の中身は README.md のファイル一覧に載せない。** README はあくまで受講者に配布する教材の索引。
- **教材本文とハンズオン出力の置き場を混同しない。** 「Skills を使ってニュースレターを作る」などのハンズオンを自分で実行する場合、成果物は `workspace/` に書き出す（ルート直下に `scale-newsletter.md` 等を作らない）。
- **教材の構造は「ハンズオンのプロンプト → 何が起きているかの解説 → スキルの中身の分解」の順** で書かれている。新しいハンズオンを追加するときも同じ流れを踏襲する。
- **README.md はファイル構成の表のみ**に留める運用。「進捗」「更新履歴」「日付」などは書かない（グローバル規約）。
- **教材内で Skill・Plugin・MCP・Subagents の使い分けを説明している箇所** は相互に整合している必要がある。片方だけ更新すると矛盾が出やすいので、用語や分類を変えるときは横断で直す。

## 受講者目線の言葉遣い

受講者は Git / ターミナル / プログラミングに不慣れな前提。技術用語を使うときは必ず日本語の補足か具体例を添える。教材内で既に確立している表記（例：「ターミナル」「スキル」「プラグイン」「サブエージェント」等のカタカナ）を勝手に英語や別表記に変えない。
